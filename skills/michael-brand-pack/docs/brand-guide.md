# Operator Notes Brand Guide

## Brand name

**Operator Notes**

## Tagline

**Field notes refined into decision briefs.**

## Author identity

Use:

- **Michael F. Bryan** for formal/public artefacts.
- **Michael-F-Bryan** for GitHub-adjacent, internet-native, and technical contexts.
- **MFB** only as a small internal mark or shorthand.

Do not overuse initials, fake terminal motifs, or personal-logo-heavy styling. The artefact system should be recognisable through structure and judgement, not decorative branding.

## Brand concept

**Operator-grade communication.**

This means the artefacts should feel like they were produced by someone close to the work who understands both technical detail and operational consequences.

They should be:

- useful
- legible
- structured
- direct
- evidence-aware
- decision-oriented
- accessible by default

## Positioning

Readable, decision-oriented artefacts for technical work that needs clarity, judgement, and forward motion.

More expressive version:

> Field notes refined into briefings: practical technical communication that makes complex work legible, exposes the trade-offs, and helps people decide what to do next.

Internal rule:

> Make the work legible. Make the next move obvious.

## Reader reaction

A successful Operator Notes artefact should make the reader think:

- “This person understands the problem.”
- “The important structure is visible.”
- “The judgement is grounded.”
- “The trade-offs are clear.”
- “I know what should happen next.”

## Brand traits

| Trait | Meaning | Avoid |
|---|---|---|
| Legible | Easy to read, scan, and parse | Faint, tiny, low-contrast UI |
| Structured | The thinking has visible shape | Wall-of-text notes, pretty chaos |
| Decisive | Recommendations are clear | Endless caveats, false neutrality |
| Grounded | Claims connect to evidence and constraints | Hype, abstraction, theatre |
| Operational | Oriented toward action and real use | Academic completeness, design vanity |

## Pillars

### 1. Legibility is authority

If the work is hard to read, it loses credibility.

Rules:

- Important text must be clearly readable.
- Borders must be visible when they carry structure.
- Metadata must not be tiny or faint.
- Dense UI is acceptable; cramped UI is not.

### 2. Structure before decoration

The layout should expose the thinking.

Useful recurring sections:

- Current read
- Recommendation
- Evidence
- Trade-offs
- Risks
- Assumptions
- Open questions
- Next move

### 3. Useful beats impressive

The artefact should not look like it is trying to win a design award. It should look like it was made by someone who cared about the reader understanding the work.

Avoid:

- decorative gradients
- visual effects without purpose
- generic card grids
- over-designed covers
- fake dashboard complexity

### 4. Calm technical confidence

The artefact should sound like someone who has done the work.

Use:

- direct sentences
- specific nouns
- visible trade-offs
- precise uncertainty
- recommendations near the top

Avoid:

- hype
- timid hedging
- performative harshness
- vague strategy language

### 5. Decisions over documentation

Do not merely describe the situation. Improve the reader’s ability to act.

Most artefacts should include:

- the current position
- the recommendation
- why it matters
- trade-offs
- risks
- next action

### 6. Accessible by default

Accessibility is not a badge. It is a baseline.

Rules:

- Primary text should target strong contrast.
- Secondary text should still be readable.
- Colour must not be the only signal.
- Interactive elements need visible focus states.
- Documents should print/export cleanly when possible.

## Visual mood

Readable technical document + field manual + clean software interface.

Avoid:

- SaaS landing page
- academic PDF
- cyberpunk terminal
- Notion template
- startup pitch deck
- corporate strategy report

## Colour direction

Light theme is canonical for reports and documents. Dark theme is secondary for tools, dashboards, demos, and long-running screen work.

Use colour for signal:

- accent: links, selected states, key highlights
- warning: risk/caution
- error: blocker/failure
- success: confirmed/done
- neutral: structure and surfaces

Do not spray colour around to make things “pop”.

## Typography direction

Recommended body/UI fonts:

- Atkinson Hyperlegible
- Inter
- IBM Plex Sans
- Source Sans 3
- system-ui fallback

Recommended monospace fonts:

- JetBrains Mono
- IBM Plex Mono
- Cascadia Code
- Roboto Mono

Do not use monospace as body text.

## Recognition system

Artefacts should be recognisable through:

1. structure
2. tone
3. recurring decision patterns
4. diagrams
5. component names
6. typography and spacing
7. colour palette

Colour should support recognition, not carry it alone.

## Anti-patterns

### Too corporate

- vague strategy language
- blue-and-white consultancy palette
- stock icons
- executive-deck gloss
- “unlock value” / “drive alignment” language

### Too casual

- jokes in structural headings
- handwritten fonts
- meme energy
- loose reasoning
- unsupported assertions

### Too developer aesthetic

- dark terminal theme everywhere
- monospace body text
- neon green/amber on black
- fake command prompts
- code metaphors where they do not belong

### Too academic

- abstract-first prose
- excessive caveats before the recommendation
- dense paragraphs
- passive voice
- completeness over action

### Too generic

- default Tailwind palette
- same card grid as every AI dashboard
- decorative icons
- headings that could belong to any company
- no recognisable decision structure
