What's the rule?

Keep the SUT real

The thing under test should mostly stay real. Replace only awkward external edges.

Assert outcomes

Persisted state, return values, files, CLI exit codes — not internal call order.

Heavy mocking = design feedback

Three mocks and a monkeypatch stack usually means fix the seam, not the mock.

What stays real?

Replace boundaries honestly. The core logic under test stays solid; edges are where fakes and mocks live.

Good vs bad test

Each pair shows one smell and a behaviour-shaped alternative. Canonical home for these examples — link here from other sections.

Pair 1 — mock theatre vs outcome

✕ Mock theatre

validator = MagicMock()
service.create_user("michael")
validator.validate.assert_called_once()

✓ Real behaviour

repo = InMemoryUserRepo()
result = service.create_user(" Michael ")
assert repo.get(result.id).name == "michael"

Pair 2 — monkeypatch soup vs explicit seams

✕ Monkeypatch soup

monkeypatch.setattr(module, "repo", fake)
monkeypatch.setattr(module, "clock", fake)

✓ Explicit seams

service = Service(
    repo=InMemoryRepo(),
    clock=FakeClock("2026-07-05"),
)

Before you mock

Work through these before adding another MagicMock.

  1. Am I replacing an external boundary, or the thing I should be testing?
  2. Could I use the real implementation cheaply?
  3. Would a small in-memory fake be clearer?
  4. Am I asserting an observable outcome, or just that calls happened?
  5. Would this test survive a sensible refactor?
  6. Is this mock hiding a design problem?
  7. Is the dependency visible in the API, or am I monkeypatching?

The ideal test gives real confidence, is boring to read, and would still pass after a sensible refactor.
