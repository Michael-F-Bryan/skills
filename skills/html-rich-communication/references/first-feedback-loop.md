# First-feedback exception

First feedback is an exception for a genuinely branching communication problem, not a default stage before HTML.

Use it only when all are true:

1. a reader, required decision, trust boundary, privacy boundary, or central claim is unknown;
2. the answer would produce materially different arguments or artefact shapes; and
3. a reasonable assumption cannot preserve truth, safety, and the user's requested outcome.

High stakes, design review, polish, an unvalidated number, an open-loop risk, or a same-day deadline do not trigger this exception by themselves. Put known uncertainty and caveats into the artefact and continue.

## Smallest useful calibration

Ask at most one focused question. Prefer ordinary chat over a file or separate artefact.

If the branch must be made visible, show only:

- assumed reader and action;
- the two materially different spines;
- what each would cut; and
- the one answer needed to choose.

Do not create a communication-contract file, assumptions ledger, content-disposition table, metric-provenance table, reviewer questionnaire, or feedback ledger unless the user asks for a durable review record. Once the branch is resolved, return immediately to the smallest applicable implementation lane.

## Stop condition

Pause before implementation only when choosing a default could mislead the reader, expose private material, or answer the wrong decision. Otherwise state the narrow assumption and continue.
